# 剧本 YAML Schema 设计文档

> 版本：1.0 | 作者：AI Novel-to-Script 工具

---

## 1. Schema 概览

本 Schema 定义了一个**面向影视/舞台剧本的 YAML 数据格式**，用于将小说文本转换为结构化剧本初稿。

### 设计原则

1. **可读性优先**：YAML 比 JSON 更接近自然语言，便于作者直接编辑
2. **扁平 + 嵌套平衡**：顶层元信息扁平化，场景内容嵌套结构化
3. **向后兼容**：所有字段均可选，新增字段不影响已有解析
4. **影视行业惯例对齐**：仿照 Final Draft / Fountain 的场次-对白-动作三段式

---

## 2. 完整 Schema 定义

```yaml
# ============================================================
# 顶层结构：一部完整的剧本
# ============================================================
script:
  # ---- 元信息 ----
  meta:
    title: "剧本标题"                    # 必填
    original_novel: "原著小说名"          # 选填
    author: "原作者"                      # 选填
    adaptor: "改编者"                     # 选填
    version: "1.0"                        # 版本号
    created_at: "2026-06-05T12:00:00Z"   # ISO 8601 时间戳
    genre: "古装 / 都市 / 科幻 / ..."     # 题材标签
    total_acts: 3                         # 幕数
    total_scenes: 12                      # 场次数
    summary: "一句话剧情梗概"             # 必填

  # ---- 角色表 ----
  characters:
    - id: "char_001"
      name: "角色名"                      # 必填
      aliases: ["别名1", "别名2"]         # 小说中可能出现的别称
      role_type: "主角 / 配角 / 反派 / 龙套"
      age: 28
      gender: "男 / 女"
      personality: "性格描述（简要）"
      background: "角色背景（1-3句话）"
      relationships:                     # 与其他角色的关系
        - target: "char_002"
          relation: "父子 / 恋人 / 仇敌 / ..."
          description: "关系简要说明"
      first_appearance: "scene_003"       # 首次出场场次
      arc_summary: "角色弧光简述"         # 角色在剧本中的成长变化

  # ---- 幕结构 ----
  acts:
    - act_number: 1
      title: "第一幕标题"                 # 如 "开端" / "建置"
      summary: "本幕剧情概要"
      scenes:
        - scene_id: "scene_001"
          scene_number: 1
          heading: "场次标题"             # 如 "第1场 内景 丞相府书房 日"
          location: "场景地点"
          time_of_day: "日 / 夜 / 黄昏 / 黎明"
          interior: true                  # true=内景 false=外景
          summary: "本场剧情概要（1-2句话）"
          characters_present:             # 出场角色
            - "char_001"
            - "char_002"

          # 场次内容：按时间顺序排列的 beats
          beats:
            # --- 动作/描写 ---
            - type: "action"
              description: "环境描写或角色动作的描述性文字。"

            # --- 对白 ---
            - type: "dialogue"
              speaker: "char_001"         # 说话人 ID（必填）
              line: "这是角色说的台词。"   # 必填
              parenthetical: "低声"       # 括号提示（表情/动作），选填
              tone: "愤怒 / 悲伤 / 平静 / ..."  # 选填

            # --- 转场 ---
            - type: "transition"
              style: "切入 / 淡出 / 溶解 / 切黑"
              description: "转场效果说明"

            # --- 旁白 / 画外音 ---
            - type: "voiceover"
              speaker: "char_001"
              line: "画外音内容"

            # --- 蒙太奇 ---
            - type: "montage"
              description: "蒙太奇段落描述"
              shots:                      # 蒙太奇中的镜头组
                - "镜头1：xxxx"
                - "镜头2：xxxx"

  # ---- 关键道具 / 场景元素 ----
  props:
    - id: "prop_001"
      name: "道具名"
      description: "道具描述"
      first_appearance: "scene_002"
      significance: "道具在剧情中的意义"

# ============================================================
# 注：所有字段均遵守以下省略规则
# - null 字段可省略整行
# - 空列表 [] 可省略整个 key
# - 未使用字段不建议输出，减少噪音
# ============================================================
```

## 3. 设计原因说明

### 3.1 为什么选择 YAML 而非 JSON？

| 维度 | YAML | JSON |
|------|------|------|
| **可读性** | 类似自然语言，无需引号和花括号 | 大量 `{}` `"` 干扰阅读 |
| **可编辑性** | 作者可直接用文本编辑器修改 | 需要小心嵌套括号 |
| **注释支持** | `#` 原生支持注释 | 无注释支持 |
| **多行文本** | `|` 和 `>` 原生支持 | 需 `\n` 转义 |
| **行业习惯** | Final Draft / Fountain 均为类 YAML 格式 | 更适合机器通信 |

**核心原因**：剧本是一个"人机协作"的产物——AI 生成初稿后，作者需要反复修改。YAML 让修改门槛降到最低。

### 3.2 为什么设计 `beats` 而非传统的"动作-对白-动作"平铺？

传统剧本格式将动作和对白交替排列。但在 AI 转换场景中，这种结构有两个问题：

1. **AI 难以精确判断交替边界**——一段动作可能对应多句对白，反之亦然
2. **结构化程度不够**——如果想让作者批量修改所有对白，平铺格式难以筛选

`beats` 数组用 `type` 明确标记每个节拍的性质，允许：
- 每种节拍携带各自专属字段（对白有 `speaker`/`tone`，动作没有）
- 脚本工具可以对特定类型做批量操作（"把所有内心独白改成旁白"）
- AI 输出时按 type 独立生成，减少幻觉

### 3.3 为什么角色表独立于场次？

| 设计 | 优点 |
|------|------|
| 角色表在顶层 `characters` | 全剧视角，一次性了解所有角色 |
| 场次仅引用 `char_001` ID | 避免每个场次重复角色信息 |
| `aliases` 字段 | 解决小说中同一角色多个称呼的问题 |

**核心原因**：小说中一个角色可能有多种称呼（"林冲"、"林教头"、"豹子头"），AI 需要建立别名映射才能准确识别对白归属。这个映射表就是 `aliases`。

### 3.4 Beats 类型的设计考量

| Beat 类型 | 来源 | 设计原因 |
|-----------|------|---------|
| `action` | 小说描写/叙述 | 涵盖环境、动作、心理描写 |
| `dialogue` | 引号内对话 | 剧本核心，需说话人 + 语气标注 |
| `transition` | 场景切换 | 影视特有，小说中常为段落分隔 |
| `voiceover` | 内心独白/叙述 | 小说第一人称常见，剧本需单独标记 |
| `montage` | 时间压缩段落 | 小说"转眼三年过去..."类描述 |

`tone`（语气）字段对于剧本**极其重要**。同一句台词"你走吧"在不同语气下（愤怒/悲伤/冷漠）是三个完全不同的场景。AI 可以从小说上下文推断语气并标注。

## 4. Schema 演进计划

| 版本 | 计划 |
|------|------|
| v1.0 | 基础剧本结构，支持对白、动作、转场 |
| v1.1 | 增加灯光/音效/镜头方向（适合拍摄） |
| v2.0 | 多线程叙事支持（并行时间线的多个场景） |
| v3.0 | 分镜脚本能力（景别、机位、运镜） |

---

> 本 Schema 的设计目标是**让小说作者拿到就能改，让制片人看到就能用**。
